# AI 使用说明

## AI 在哪些关键地方提供了帮助

- **需求拆解**：辅助分析作业文档，梳理核心问题和系统边界
- **技术选型对比**：对 PHP/Laravel、Go、Java/Spring Boot 的优劣进行了横向对比分析
- **架构设计**：提出了 HTTP API + SQLite Job Queue + Worker 的整体架构方案
- **代码实现**：生成了 HTTP Handler、Worker Dispatcher、SQLite 持久化层等主要代码
- **项目结构规划**：建议了 `internal/` 分层结构（handler/worker/store/model）
- **供应商抽象方案分析**：针对"多供应商、不同 API 格式"的需求，AI 梳理了 4 种演进方案（保持透传 / 配置+模板 / 代码 Adapter / 混合模式），并对每种方案的适用场景、优缺点做了对比分析，帮助我快速理解取舍空间
- **Vendor Adapter 架构实现**：在我确定 B+C 组合方案后，AI 生成了完整的 vendor 包（接口定义、配置驱动 Adapter、Registry）、Store 层 Vendor CRUD、Handler 层供应商管理 API，以及 21 个覆盖核心链路的单元测试
- **幂等去重实现**：在我决定加入 `biz_id` 去重机制后，AI 完成了全链路改造——数据模型新增字段、SQLite 唯一部分索引、Store 层 INSERT 冲突处理（返回已有记录）、Handler 层区分 202/200 响应、以及配套的幂等测试用例
- **默认供应商预置**：AI 实现了 `SeedDefaultVendors` 方法（INSERT OR IGNORE 语义），以及启动流程中的自动调用
- **API 响应格式分析**：我提出通知接口的返回值不符合工程规范后，AI 梳理了通知 API 与 Vendor API 的响应格式差异（Create 有无 message 包装、列表 key 不统一、Get 裸返回实体等），并给出了 3 种改进方向供我选择
- **统一响应封装实现**：在我决定采用统一信封方案后，AI 完成了全量改造——新增 `model.Response` / `model.ListData` 结构体、`handler/response.go` 辅助函数（`respondSuccess` / `respondError` / `respondList`）、重构通知和 Vendor 全部 handler、适配测试用例、同步更新 ARCHITECTURE.md 和 README.md 文档（每个接口补充完整的请求参数表和响应参数表）

## AI 给出过哪些我没有采纳的建议

- **引入 MySQL/PostgreSQL 作为存储**：AI 最初推荐使用关系型数据库做 Job Queue。我认为 MVP 阶段过重，增加了部署和运维成本，改为 SQLite
- **引入 Redis 作为队列**：AI 建议用 Redis 提升性能。我判断当前阶段无必要，引入额外进程增加了运维复杂度
- **使用 Gin 框架**：AI 建议用 Gin 做 HTTP 路由。实际下载依赖时发现 Gin 引入了大量间接依赖（protobuf、yaml、sonic 等），与控制复杂度的原则不符，最终改用 Go 标准库 `net/http`
- **使用 modernc.org/sqlite（纯 Go SQLite）**：AI 建议使用纯 Go 实现的 SQLite 驱动以避免 CGo 依赖。实际下载时发现包体积过大、下载极慢，改为 `mattn/go-sqlite3`
- **供应商配置化管理（方案 B）**：AI 在第一版中建议预先注册供应商、业务系统只传事件类型。我当时认为这超出了第一版的系统边界，选择了完全透传的方案 A。在第二轮沟通中我推翻了这个决策（详见下方）
- **混合模式（方案 D）**：AI 推荐"混合模式"——新请求走 vendor 配置，旧请求继续走透传，向后兼容。我认为透传模式本身就是错误方向，没必要兼容，直接全部切到 vendor 驱动模式
- **响应格式轻量规范化（方案 B）**：AI 提出不引入封装类型，仅约定成功时直接返回实体或 `{data, count}`、错误时 `{error, code}` 的轻量方案。我认为这只是打补丁，没有从根本上解决格式不统一的问题
- **仅对齐通知和 Vendor 格式（方案 C）**：AI 提出最小改动方案——让通知 API 对齐 Vendor 的返回风格。我认为既然要改就应该一步到位，引入统一信封比逐接口对齐更有长期价值

## 哪些关键决策是我自己做出的，以及原因

- **选择 Go 语言**：在 PHP 和 Go 之间权衡后，选择 Go——它能更清晰地展示每一个工程决策（队列、重试、持久化），而不是被框架的默认行为遮掩
- **存储选 SQLite**：主动拒绝了 AI 推荐的重型数据库，判断依据是 MVP 阶段应优先降低运维成本和部署门槛
- **标准库替代 Gin**：在实践中发现 Gin 的依赖树过于庞大，主动决定改用标准库，这是在实现过程中做出的工程权衡
- **时间戳用 Unix 整数存储**：遇到 SQLite DATETIME 文本格式与 Go 驱动之间的解析兼容问题后，选择改用 Unix 时间戳（INTEGER），彻底消除格式解析歧义
- **推翻"不做幂等去重"，改为 biz_id 去重**：第一版认为幂等应由接收方保证，本系统不介入。重新审视后发现：业务方可能因网络超时等原因重复提交相同通知，如果系统不去重就会产生大量重复投递，浪费资源并增加接收方压力。因此引入 `biz_id` 作为业务去重标识，通过 `(vendor_id, event, biz_id)` 唯一索引在数据库层面保证提交侧幂等。投递侧的幂等仍由接收方保证
- **不做 Token 自动刷新**：外部 API 的认证由调用方管理，本系统只负责「忠实投递」
- **重试上限后人工介入而非无限重试**：无限重试可能造成资源浪费和对外部系统的持续压力，死信 + 手动重放是更合理的工程选择
- **推翻第一版透传方案，改为 B+C 组合**：第一版选择了"完全透传"（调用方自己拼 URL/Headers/Body），在重新审视需求后认为这是错误的。核心原因：（1）调用方不应关心供应商 API 的任何细节，只需要告知"发生了什么事"；（2）供应商的 API 密钥、请求格式等配置应由通知系统集中管理，而非散落在各个业务系统中；（3）供应商会不断增加，必须有可扩展的抽象。最终选择方案 B（配置+模板驱动）和方案 C（代码级 Adapter）组合：大多数供应商通过 DB 配置零代码接入，复杂签名/加密场景用代码 Adapter 扩展。Registry 按"代码 Adapter 优先、配置 Adapter 兜底"的顺序解析
- **引入统一响应封装，而非打补丁**：发现通知 API 和 Vendor API 的响应格式不一致（通知 Create 用 `{message, job}` 包装，Vendor Create 裸返回实体；列表 key 各自命名），AI 给出了三种方案：A（统一信封）、B（轻量约定）、C（最小对齐）。我选择方案 A——引入 `Response{code, message, data}` 统一信封，理由是：既然要改就一步到位，统一格式比逐接口修补更有长期价值，调用方只需解析一种结构。同时要求文档中每个接口都给出完整的请求参数和响应参数说明
